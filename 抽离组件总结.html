一、普通业务组件

通过props传值来满足差异化需求

提升开发效率，同时复用ui和逻辑，使用方便
缺点:
1，不够灵活，经常需要修改已封装好的组件代码来满足新场景，组件复用性差
2，代码中充满着过多的type判断（if else，switch，匹配对象），代码可维护性差

二、HOC
官方定义：高阶组件是参数为组件，返回值为新组件的函数

优点：
1，抽离共用逻辑，其他组件通过HOC包装后，能获得共用逻辑
2，能够对传入的组件的渲染做增强和拦截

缺点:
1，过多的嵌套，在业务复杂的页面，维护艰难
可解决的缺陷：丢失静态方法，丢失displayname, 最好手动维护一下

示例：https://ai.ifchange.com/ability/purpose https://ai.ifchange.com/ability/skill
https://gitlab.ifchange.com/micro-ai/ai-index/blob/xl-share/src/pages/ability/components/UseFetchDataHook/index.js

三、Render Props
https://zh-hans.reactjs.org/docs/render-props.html

官方定义：指一种在 React 组件之间使用一个值为函数的 prop 共享代码的简单技术，是一个用于告知组件需要渲染什么内容的函数 prop

优点：
1，数据流向更直观了，子孙组件可以很明确地看到数据来源，可以使用带有 render prop 的常规组件来实现大多数高阶组件 (HOC)

与HOC对比：
Render Props 通过父组件将可复用逻辑封装起来，并把数据(或dom)提供给子组件。至于子组件拿到数据之后要怎么渲染，完全由子组件自己决定，灵活性非常高。而高阶组件中，渲染结果是由父组件决定的。Render Props 不会产生新的组件，children prop更加直观的体现了「父子关系」

React.cloneElement类似处理, 需要注意父子关系

示例：
1，传递状态：
https://gitlab.ifchange.com/micro-ai/ai-index/blob/xl-share/src/pages/ability/components/UseFetchDataHook/index.js
在大部分情况下，高阶组件和 Render Props 是可以相互转换的，也就是说用高阶组件能实现的，用 Render Props 也能实现。只不过在不同的场景下，哪种方式使用起来简单一点罢了。

四，自定义Hook
自定义 Hook 是一个函数，其名称以 “use” 开头，函数内部可以调用其他的 Hook；

确保只在自定义 Hook 的顶层无条件地调用其他 Hook

示例：（https://ai.ifchange.com/ability/purpose）
https://gitlab.ifchange.com/micro-ai/ai-index/blob/xl-share/src/pages/ability/components/UseFetchDataHook/index.js
建议：
1，自定义 Hooks 的返回值可以使用数组类型，更易于在外部重命名。

高阶组件和 Render Props 本质上都是将复用逻辑提升到父组件中，而 Hooks 出现之后，我们将复用逻辑提取到组件顶层，而不是强行提升到父组件中。这样就能够避免 HOC 和 Render Props 带来的「嵌套地狱」

自定义Hook 会替代 render props 和高阶组件吗？（https://zh-hans.reactjs.org/docs/hooks-faq.html#do-hooks-replace-render-props-and-higher-order-components）
官方回答：通常，render props 和高阶组件只渲染一个子节点。我们认为让 Hook 来服务这个使用场景更加简单。但在大部分场景下，Hook 足够了，并且能够帮助减少嵌套。

五、Compound comopnent
将父子组件间的状态共享内置，使用者只需考虑传入的children，不用考虑如何和父组件通信，比较适应ui多变的场景
